# Cicada Temporal Fact-Graph (TFG)

## Philosophy

Stop modeling *things* as the primary unit. Model **claims about things, over time**.

The current Cicada model makes the entity page the atom: a unit of knowledge is "a page about
LEANN." But an agent almost never needs "everything about LEANN" — it needs a specific
*assertion*: "Rodrigo chose LEANN over FAISS for storage reasons," "Cicada now uses sqlite-vec
*instead of* LEANN," "the index is rebuilt nightly." Each of those is a **fact**: a typed,
timestamped, attributable edge between two lightweight nodes. Facts are where truth, change,
contradiction, provenance, and context actually live. Entities are just the join keys facts
hang off of.

The single design commitment, taken from Graphiti/Zep and sharpened for a git-backed substrate:

> **The primary unit is a bi-temporal FACT. A fact is an edge `(subject) --predicate--> (object)`
> carrying two independent time axes — when it was *true in the world* (valid-time) and when
> *Cicada learned it* (ingestion-time) — plus an `invalidated_by` pointer. Beliefs are never
> overwritten and never deleted. They are **superseded**: a new fact closes the old one's
> valid-time and points back at it. Contradiction handling becomes mechanical, not a judgment
> call at read time.**

Five consequences that directly answer Rodrigo's intuitions:

1. **The promotion gate is deleted.** There is no "2nd mention" rule. Every extracted fact is
   *appended* (Mem0-2026 ADD-only). A node referenced once is a thin stub; a node referenced
   100×accretes 100 facts. "Importance" is an emergent property of fact-density and recency,
   not a hard threshold. Anti-pollution is handled by *confidence + decay on facts*, not by
   refusing to write.
2. **Context-dependent identity is free.** A fact carries a `context` field (the RDF named-graph
   4th dimension). "Rodrigo `values` speed" in `context: engineering` and "Rodrigo `values`
   slowness/presence" in `context: family` are two non-contradictory facts on the *same* node.
   No separate `Rodrigo-engineer` page, no context collapse.
3. **Temporal change is the native operation, not a special case.** Postgres→SQLite is one new
   fact that invalidates one old fact. Both stay queryable.
4. **Procedural/preference memory is just a fact type** (`predicate: prefers`,
   `predicate: procedure_for`) with structured slots — first-class, not a bolted-on page type.
5. **The closed 8-type taxonomy is demoted.** Node `kind` becomes a soft, open hint (helpful for
   graph coloring and filtering), never a gate. The *epistemic* typing that matters
   (explicit/deductive, valid/invalid, user-stated/agent-inferred) moves onto the **fact**.

Markdown + git + sqlite-vec stays exactly as the substrate. Git is not incidental here — it is
the structural advantage Honcho lacks: the immutable fact ledger and the bi-temporal "what did
we believe on date X" query are *already* git's native semantics. We lean into that hard.

## Data model

### Layout on disk

```
memory/
├── facts/                         ← NEW: the primary store. One file per subject node.
│   ├── rodrigo.facts.yaml         ← all facts whose subject is `rodrigo`
│   ├── leann.facts.yaml
│   ├── cicada.facts.yaml
│   └── ...
├── nodes/                         ← was entities/. Thin node cards (mostly generated).
│   ├── rodrigo.md
│   ├── leann.md
│   └── ...
├── episodes/                      ← unchanged. Raw source of truth for ingestion.
├── clusters/                      ← NEW (GraphRAG community summaries), generated by Sleep.
├── vector_index.db                ← sqlite-vec, now also indexes FACTS (see below).
├── nudges/  clarifications/  inbox/  hubs/  sources/   ← unchanged
└── graph_edges.yaml               ← KEPT as a *derived* projection for the d3 graph view only.
```

Two deliberate decisions:

- **Facts are grouped by subject node, one YAML file per node** (`facts/<node-id>.facts.yaml`),
  not one file per fact. 1,882 nodes → ~1,882 fact files, each holding an ordered list of fact
  records. This keeps `git blame` meaningful (a node's belief history is one file's history),
  keeps file count bounded, and makes "load everything about X" a single read. A fact about
  `(rodrigo)->(leann)` is stored under the **subject** (`rodrigo.facts.yaml`) and the index
  records both endpoints, so traversal from either side is an index lookup.
- **`graph_edges.yaml` is retained but demoted to a derived view.** Sleep regenerates it from
  the *currently-valid* facts so the existing d3 `/graph` endpoint keeps working untouched
  during migration. It is no longer authoritative.

### A node card — `nodes/leann.md`

Thin. Mostly generated by Sleep from the facts. Human-readable, Obsidian-compatible, the
human-facing summary — but **not** the reasoning substrate.

```markdown
---
id: leann
display_name: LEANN
kind: tool                 # soft hint, open set, never a gate
status: superseded         # derived: any still-valid fact? → active; all invalidated? → superseded
created: 2026-04-10
last_fact_at: 2026-06-13    # ingestion-time of most recent fact touching this node
fact_count: 7
valid_fact_count: 2
aliases: [Leann, leann-index]
---

<!-- GENERATED by Sleep from facts/leann.facts.yaml — do not hand-edit below this line -->
## Current beliefs (valid)
- chosen-over FAISS for storage savings (conf 0.93, since 2026-04-10) [[Rodrigo]]
## Superseded
- ~~is the vector index of Cicada~~ → superseded 2026-06-13 by sqlite-vec [[sqlite-vec]]
```

The body is regenerable from facts; if it's ever lost, Sleep rebuilds it. The human edits facts
*through the app/nudges*, which write to the YAML — they don't hand-edit this card.

### A fact — the atom — in `facts/rodrigo.facts.yaml`

```yaml
node: rodrigo
facts:
  - id: f_2026-06-13_0007            # f_<ingest-date>_<seq>, globally unique, stable
    subject: rodrigo
    predicate: prefers               # open, lowercased verb phrase; Sleep normalizes synonyms
    object: split-fastapi-routers-by-domain   # node-id OR literal value
    object_kind: literal             # node | literal
    context: engineering             # engineering|family|philosophical|cross|general (open)
    statement: "Rodrigo prefers FastAPI repos with routers split one-file-per-domain."

    # --- bi-temporal axes (the heart of the model) ---
    valid_from: 2026-03-01           # when this became true in the world (best estimate)
    valid_to: null                   # null = still valid; a date = closed/superseded
    ingested_at: 2026-06-13T15:09Z   # when Cicada learned it (transaction time)
    invalidated_by: null             # fact-id that superseded this, if any
    supersedes: null                 # fact-id this one closed, if any

    # --- epistemics (orthogonal axes, per epistemics-contradiction-decay research) ---
    epistemic_status: explicit       # explicit | deductive | inductive | abductive
    source_trust: user_stated        # user_stated | agent_extracted | agent_reflected | external
    confidence: 0.9                  # 0..1, decays over time per source_trust-weighted rate
    decay_rate: 0.02                 # slow for user_stated, fast for agent_reflected

    # --- provenance ---
    source_episodes: [ep_2026-03-01_002, ep_2026-05-20_001]
    premises: []                     # for deductive facts: fact-ids it was inferred from
    authored_by: gpt-5.4-mini        # also captured as Cicada-Author git trailer
```

**Predicate** is an open vocabulary (verb phrase), not a closed enum — Sleep maintains a
`facts/_predicates.yaml` canonical-synonym map (`chose ≈ selected ≈ picked → chose-over`) so
retrieval isn't fragmented by phrasing. This is the schema-free/typed *sweet spot* from the
`typed-vs-schemafree` research: structure where it pays (the four temporal+epistemic axes are
rigid), freedom where it doesn't (the predicate phrasing).

**Why this is faceted identity for free:** `rodrigo.facts.yaml` holds, simultaneously and
without contradiction, `(rodrigo)-values->speed [context: engineering]` and
`(rodrigo)-values->presence [context: family]`. The node is one join key; the facets are
context-tagged facts. The Sleep cross-link stage looks for abstract bridges *across* contexts
(see Worked examples #1).

### How the sqlite-vec index changes

Today the index has passages keyed to `entity_id`. We **add a facts table** so retrieval is
over claims, not pages:

```sql
-- existing: nodes(id, passage, embedding)  [from node cards]
CREATE TABLE facts (
  fact_id        TEXT PRIMARY KEY,
  subject        TEXT, predicate TEXT, object TEXT, context TEXT,
  statement      TEXT,                 -- the embedded text
  valid_from     TEXT, valid_to TEXT,  -- NULL valid_to = currently valid
  ingested_at    TEXT,
  epistemic_status TEXT, source_trust TEXT, confidence REAL,
  invalidated_by TEXT
);
CREATE VIRTUAL TABLE facts_vec USING vec0(embedding float[768]);  -- statement embedding
CREATE INDEX idx_facts_subj ON facts(subject);
CREATE INDEX idx_facts_obj  ON facts(object);
CREATE INDEX idx_facts_valid ON facts(valid_to);  -- cheap "currently-valid" filter
```

The index stays **derived and disposable** (the existing `vector_index.py` contract): Sleep
rebuilds `facts`/`facts_vec` from the YAML. The embedding model (local EmbeddingGemma-300M,
768-dim) is unchanged; we just embed `statement` strings instead of (or in addition to) node
bodies. Retrieval becomes **hybrid over facts**:

1. ANN over `facts_vec` on the query → candidate fact-ids.
2. SQL filter: by default `valid_to IS NULL` (only currently-true facts) unless the caller asks
   for history; optional `context = ?`, `subject = ?`.
3. Score = `cosine × confidence × recency(ingested_at)` — the Generative-Agents three-signal
   formula, applied to facts. (Recency from `ingested_at`, importance proxied by `confidence`.)
4. Graph expansion: pull facts sharing a subject/object with top hits (1-hop traversal via the
   `idx_facts_subj/obj` indexes) for relational depth.

### How Sleep writes it (5 stages, re-mapped — no new stage count)

- **Stage 1 — Fact extraction.** LLM reads unprocessed episodes and emits *facts* (the YAML
  record above), not entity pages. Prompt asks for `subject, predicate, object, context,
  statement, valid_from, epistemic_status, source_trust`. No promotion gate — every fact is a
  candidate write.
- **Stage 2 — Node & predicate resolution.** Fuzzy + embedding + LLM disambiguation maps
  subject/object strings to existing node-ids ("Mongo"→`mongodb`), and normalizes predicate
  phrasing against `_predicates.yaml`. Creates thin node stubs for genuinely new join keys
  (cheap — just an id + kind hint).
- **Stage 3 — Contradiction = invalidate-and-supersede.** This is now the *core* stage. For
  each new fact, find existing valid facts with the same `(subject, predicate, context)`. If the
  object differs and the relation is single-valued (Sleep asks the LLM "can both be true at
  once?"), set the old fact's `valid_to = new.valid_from`, `invalidated_by = new.id`, and
  `new.supersedes = old.id`. **Nothing is deleted.** Multi-valued relations (e.g. `uses`)
  coexist. Temporal decay also runs here: confidence drops on facts not re-observed, weighted by
  `source_trust` and original `decay_rate`.
- **Stage 4 — Pattern/skill facts + cross-context links.** Recurring interaction patterns become
  `prefers`/`procedure_for` facts (procedural memory). The cross-context abstraction pass looks
  for the *same predicate across different contexts* on the same node and emits `deductive`
  bridge facts (e.g. "Rodrigo optimizes-for-the-essential in both engineering and family
  contexts" — `epistemic_status: abductive`, low confidence, flagged for a clarification nudge).
- **Stage 5 — Regenerate + commit.** Regenerate node cards, `graph_edges.yaml` (valid facts
  only), and `clusters/` community summaries. Rebuild `facts` / `facts_vec`. Commit with
  structured message; `authored_by` on each new fact becomes a `Cicada-Author:` trailer via the
  existing `build_commit_message(...)`. **Crucially: a superseding edit touches one YAML file
  and closes one fact — `git blame facts/leann.facts.yaml` now tells the exact commit, episode,
  and model that flipped a belief.**

### How /ask + MCP retrieve it

`/ask` keeps its current shape (answer + citations + confidence + gaps) but:
- Citations point at **facts** (`fact_id` + statement + valid window + source_trust), not just
  entity pages — far more auditable. "I believe X (conf 0.9, user-stated, valid since 2026-03,
  ep_2026-03-01_002)."
- The prompt is fed only **currently-valid** facts by default, plus, when the query is
  historical ("what *did* I use before sqlite-vec?"), the superseded chain.
- `confidence` aggregates from the fact-level confidences of the cited facts (min/mean), so a
  thin/decayed evidence set honestly produces low confidence + a populated `gaps`.
- MCP Bookworm's progressive disclosure becomes: cluster summary → relevant facts → node card →
  source episode. The `(observer=cicada, observed=rodrigo, context=…)` framing maps onto a
  `context`-filtered fact query — "who is Rodrigo *right now in this engineering session*" is
  `SELECT ... WHERE subject='rodrigo' AND context IN ('engineering','cross') AND valid_to IS NULL`.

## Worked examples

### 1. Engineer-self vs family-self (faceted identity, no context collapse)

Same node `rodrigo`, three coexisting valid facts in `facts/rodrigo.facts.yaml`:

```yaml
- {id: f_..._01, subject: rodrigo, predicate: values, object: shipping-speed,
   context: engineering, statement: "In engineering, Rodrigo values shipping fast and iterating.",
   valid_from: 2025-09-01, valid_to: null, source_trust: agent_extracted, confidence: 0.8}
- {id: f_..._02, subject: rodrigo, predicate: values, object: presence-and-slowness,
   context: family, statement: "With family, Rodrigo values being present and unhurried.",
   valid_from: 2025-09-01, valid_to: null, source_trust: user_stated, confidence: 0.9}
- {id: f_..._03, subject: rodrigo, predicate: optimizes-for, object: the-essential,
   context: cross, epistemic_status: abductive, premises: [f_..._01, f_..._02],
   statement: "Across contexts Rodrigo strips to the essential — fast where it ships value,
               slow where presence is the value.", confidence: 0.5, source_trust: agent_reflected}
```

No contradiction (different `context`). Fact `_03` is the **abstract cross-link** Rodrigo wants —
generated in Stage 4, low-confidence, flagged as a clarification nudge ("Does this resonate?").
A `context: family` agent session retrieves `_02` and `_03` but not `_01`.

### 2. A belief that changes — Postgres → SQLite

Day 1 (`facts/cicada.facts.yaml`):
```yaml
- id: f_2026-01-10_004
  subject: cicada, predicate: uses, object: postgres, context: engineering
  statement: "Cicada's storage layer uses Postgres."
  valid_from: 2026-01-10, valid_to: null, ingested_at: 2026-01-10T..., confidence: 0.85
```
Day N, an episode says "switched to SQLite." Stage 3 detects same `(cicada, uses-storage,
engineering)`, single-valued, contradicts → it closes the old fact and writes the new one:
```yaml
- id: f_2026-01-10_004    # MUTATED in place — only the closing fields change
  ...
  valid_to: 2026-02-02, invalidated_by: f_2026-02-02_011
- id: f_2026-02-02_011
  subject: cicada, predicate: uses, object: sqlite, context: engineering
  statement: "Cicada switched its storage layer from Postgres to SQLite."
  valid_from: 2026-02-02, valid_to: null, supersedes: f_2026-01-10_004, confidence: 0.9
```
`/ask "what DB does Cicada use?"` → SQLite (valid). `/ask "what did Cicada use before?"` →
follows `supersedes` → Postgres until 2026-02-02. `git blame` on the file → the exact commit,
episode, and authoring model that flipped it. Zero data lost.

### 3. A procedural preference

```yaml
node: rodrigo
facts:
  - id: f_2026-06-13_0007
    subject: rodrigo, predicate: procedure_for, object: fastapi-repo-layout
    context: engineering, source_trust: user_stated, confidence: 0.9, decay_rate: 0.01
    statement: "When scaffolding a FastAPI repo, Rodrigo wants routers split one-file-per-domain
                under routers/, services/ for logic, models/ for pydantic, thin main.py."
    valid_from: 2026-03-01, valid_to: null
    steps:                              # optional structured slot for procedures (Voyager-style)
      - "Create routers/<domain>.py per bounded context"
      - "Put business logic in services/, not routers"
      - "Keep main.py to app wiring + router includes"
```
At conversation start, MCP injects all `predicate IN (prefers, procedure_for)` valid facts as a
compact "how Rodrigo likes things done" block (always-injected, not retrieved-on-demand — the
preference-memory research finding). If Rodrigo later says "actually, group routers by feature,"
Stage 3 supersedes this exactly like example 2.

### 4. A saved media item

A saved Instagram reel / RSS article is a **node** (`kind: media`) plus facts *about* it:

```yaml
# nodes/reel-robotics-grasping.md frontmatter: kind: media, url, saved_at, title
# facts/reel-robotics-grasping.facts.yaml
- id: f_2026-05-20_022
  subject: reel-robotics-grasping, predicate: is-about, object: dexterous-manipulation
  context: engineering, source_trust: external, confidence: 0.7
  statement: "Saved reel demonstrates dexterous-manipulation grasping for legged robots."
  valid_from: 2026-05-20, valid_to: null, source_episodes: [ep_2026-05-20_media]
- id: f_2026-05-20_023
  subject: reel-robotics-grasping, predicate: relevant-to, object: robo-striker
  context: engineering, epistemic_status: deductive, premises: [f_2026-05-20_022], confidence: 0.6
  statement: "Relevant to the Robo Striker project (both legged-robot manipulation)."
```
Media drops cleanly into the same model — `is-about`/`relevant-to` facts make it retrievable and
*linked into the graph* without any special media schema (answers C7 extensibility).

## Migration (from ~1,882 typed entity pages)

Incremental, reversible, runnable in one Sleep-style batch. **The old `entities/` dir is never
deleted until the new path is validated** — both can coexist.

**Step 0 — Freeze & branch.** New code path behind a `MEMORY_MODEL=tfg` flag. `nodes/` and
`facts/` are new dirs; `entities/` stays read-only during dual-run.

**Step 1 — Nodes (mechanical, $0).** For each `entities/<x>.md`: copy to `nodes/<x>.md`, map
`type`→`kind`, `related`/`tags` carried over, drop `confidence`/`decay_rate`/`last_referenced`
from the node card (those move to facts). Pure file transform, no LLM. ~1,882 cheap writes.

**Step 2 — Fact extraction from existing pages (the one LLM-cost step).** For each entity page,
one LLM call: "Given this entity page and its wikilinks, emit the facts it asserts." The page
body ("LEANN is the vector index… Rodrigo chose it because… lower storage than FAISS") yields
3–8 facts. `graph_edges.yaml` (every `source/target/label` stanza) converts to facts
*deterministically* ($0): `{subject: source, predicate: label, object: target}` — this alone
backfills the relational layer for free. The LLM is only needed to mine the prose bodies and to
set `valid_from`/`source_trust`/`context`. Sensible defaults for legacy facts: `source_trust:
agent_extracted`, `valid_from: created`, `context: general`, `confidence:` carried from the old
page's confidence.

**LLM cost estimate.** 1,882 pages × one extraction call. Bodies are short (the LEANN page is
~60 words; most are 1–3 sentences). At ~600 input + ~400 output tokens/page on a mini model
(gpt-5.4-mini class): ≈1.1M input + 0.75M output tokens total. **Order of a few dollars**, run
once, overnight. The `graph_edges.yaml`→facts conversion (the bulk of the relational signal)
costs nothing. Re-runnable per-batch; failures are per-page idempotent.

**Step 3 — Rebuild the index.** Run the new Sleep index build: embed all `statement` strings,
populate `facts`/`facts_vec`. Node-card embeddings stay too (keep `nodes` table for "about X"
queries). Disposable, ~one embedding pass over ~8k–12k facts.

**Step 4 — Dual-run & validate.** Point `/ask` at facts, keep `entities/` for fallback. Run the
Table-1 recall benchmark (`benchmarks.run_table1`) on both; the fact path should win on
contradiction/temporal questions and tie-or-better on recall. d3 `/graph` reads the regenerated
`graph_edges.yaml` and needs no change.

**Step 5 — Cut over.** Flip default, archive `entities/` to `entities_legacy/` (git-tracked, not
deleted). `git mv`, one commit. Fully reversible by flag.

## Scorecard (C1–C9, honest)

- **C1 Agent retrieval quality & usability — 9.** Hybrid fact retrieval returns the *exact
  claim* with its valid window + trust, not a whole page to re-read; 1-hop graph expansion gives
  traversal; the agent can tell trust from `source_trust`+`confidence`. Strongest dimension.
- **C2 Context-dependent identity — 9.** `context` on every fact = RDF named-graph power at zero
  infra; faceted self without context collapse; Stage-4 cross-context abductive links are the
  feature Rodrigo asked for. Weakness: `context` is a small open enum the LLM must apply
  consistently — drift risk needs a canonical list + Sleep normalization.
- **C3 Temporal change & contradiction — 10.** This is the model's whole point. Bi-temporal +
  invalidate-and-supersede + git is mechanical and lossless; "what did I believe on date X" is a
  native query. Best-in-class vs every framework surveyed.
- **C4 Provenance & confidence — 9.** Per-fact `source_episodes`, `premises`, `authored_by`
  (→Cicada-Author trailer), orthogonal `source_trust`/`confidence`, and `git blame` on a
  one-fact-per-line YAML give line-level provenance. Slight loss: node-level prose nuance not
  captured as a fact is only in the (regenerable) card.
- **C5 Procedural / preference memory — 9.** First-class `prefers`/`procedure_for` facts with
  optional `steps`, always-injected at session start, superseded on change. Far ahead of mem0/
  cognee/Letta. Weakness: extracting clean structured `steps` from messy chat is LLM-quality-bound.
- **C6 Implementation cost & migratability — 7.** `graph_edges.yaml`→facts is free and does the
  heavy lifting; substrate (md+git+sqlite-vec) unchanged; dual-run is safe. But it *is* a real
  schema change touching Sleep stages 1–5, `vector_index.py`, and `ask_service`, plus a
  few-dollar one-time LLM pass over 1,882 pages. Honestly the most expensive candidate to build.
- **C7 Extensibility — 9.** New knowledge kinds = new node `kind` + new predicates, no schema
  migration (media example). Open predicate vocab + soft node kinds resist graph sprawl better
  than a closed taxonomy.
- **C8 Simplicity / maintainability — 6.** *Conceptually* simpler (one atom: the fact). But
  operationally it's more moving parts than today: bi-temporal bookkeeping, predicate
  normalization, card regeneration, a derived `graph_edges.yaml`. A solo dev must keep the
  invalidation logic correct — the riskiest code. This is the real cost of the model's power.
- **C9 OVERALL — 9.** The best *agent* memory system among plausible candidates: it nails the
  three things production frameworks fail at (temporal contradiction, context-dependent identity,
  procedural memory) while keeping the cheap, portable, git-honest substrate. You pay for it in
  build complexity (C6/C8), which is the right trade for "best memory for my agents."
